PDF Standards

More than half of the content on AAP.org is in print format (ie, PDF). If you are migrating your current site, you should start collecting PDFs as soon as you begin the process of transferring your content.

PDF vs Web Page Naming
Optimize with Acrobat

Note: AAP.org cannot accept PDFs larger than 50MB (50,000kb) in size. If you are going to submit PDFs to the document library of AAP.org, it is imperative that you first optimize the documents to fall within the size limits for the site. Optimizing content for the web is the responsibility of the content owner and a crucial part of keeping speed up and server demand down for the entire site collection. You may have to play with the settings within Acrobat to optimize the PDF, especially w/r/t images, which can greatly increase the file size of a PDF. Scanned PDFs (ie, pages converted to PDF from a copier) are also often much larger than necessary because they are saved as images rather than text documents.

PDFs account for much of the storage across the AAP.org site collection. To prepare for the the tens of thousands of anticipated PDFs on the site over the next couple years, Editorial is centralizing all PDFs in a single document library at the root (i.e., the uppermost site). To do this, we need uniform filenames and metadata. This not only allows the documents to be more human-readable, it provides extensibility across all AAP.org sites. Properly tagged and organized content means we can eventually create a more seamless and customized experience for end users.

PDF vs Web Page

Why web pages are preferred on AAP.org:

  1. PDFs do not adjust their content according to device or browser size. This defeats the purpose of a responsive web design
  2. Depending on the user's browser preference, a PDF an open within the browser (eg, on most desktop browsers) or need to be downloaded separately for a mobile device (eg, the default browser setting on some Android OS mobile devices)
  3. Even if the end user is able to open a PDF within a browser, he/she is viewing AAP.org content outside of the context of the site. This means no navigation, which precludes the user's ability to discover related content within the AAP.org environment.
  4. PDF and web pages are not mutually exclusive. In modern browsers, end users always have the ability to select File>Print>Print to PDF... and then save or print their content with a .pdf file extension. If styling is a concern, you can always include a "Download PDF" link at the top of the web page.
  5. It is easier to read text when fonts are adjusted to screen size rather than forcing the end-user to zoom in or out to read the text of your PDF.
  6. File size for PDFs are often much larger than web pages. This means more demand on our servers and more data usage for our users.
  7. Because PDFs are often downloaded or opened in a separate window rather than viewed within the AAP.org environment, it is easier to forget about the content stored within the document body. This usually means that content in static PDFs is not updated as often as a regular web page. Also, updating may prove more time-consuming once content owners make changes to their own sites, since changing simple copy in the Sharepoint CMS is faster than opening and editing a PDF file in a separate application.

When PDFs are A-okay:

  1. PDF documents that are longer than 5 pages in length, although sometimes longer documents can still be converted to a web page if the content can be collapsed or expanded by the end user. Here is an example from the coding subsite.
  2. PDFs that are especially rich in relevant photography or that include extensive layout or design may not transfer over well to the website. However, it may be worth it to create a page that summarize the content of the PDF with a "Download the PDF" link included in the page body.

Naming PDFs

Note: This same information can be found in naming conventions for PDFs.

PDFs should take the form of the naming convention abbreviation followed by an underscore followed by a short, consistent description:

  • federaladvocacy_chip_principles.pdf
  • federaladvocacy_chip_templatelettertogovernors.pdf
  • 9.12.14combinedfederalupdate.pdf

Moving forward, all PDFs are going to be stored at the root. This means that if you give you name your PDF according to the new naming convention, you will have a permanent link for that document to use in the communication medium of your choice. Just prepend "http://www.aap.org/en-us/Documents/" to your PDF and you're set. Continuing with the above examples:

  • http://www.aap.org/en-us/Documents/federaladvocacy_chip_principles.pdf
  • http://www.aap.org/en-us/Documents/federaladvocacy_chip_templatelettertogovernors.pdf

Submit your PDFs via the site requests at AAP.org/websites or find out how to submit new content.

Some content types are common enough across the site collection that they merit more extensive naming conventions.

Newsletter filenames begin with the site abbreviation from above. However, unlike most PDFs, newsletters can—and should—include a chronological reference. The format is either volume-issue (vv_ii) or just volume/season/month, depending on frequency of publication:

  • abbreviation_newsletter_yyyy_vv_ii
  • socc_newsletter_2014_04_01
  • sohm_newsletter_2012_summer
  • newsletter_FINAL
  • Newsletter_SOHM_july7-07-14
  • sohm_7-4-14_newsletter

The same rule applies for fact sheets (another common content type in the CMS), with the addition of spelling out the full state name:

  • abbreviation_factsheet_location
  • federaladvocacy_medicaidfactsheet_illinois
  • federaladvocacy_medicaidfactsheet_IL
  • medicaidfactsheet_IL
  • medicaidfactsheet_illinois

That said, yearly fact sheets might not merit including a year in the filename. For example, if an outside user wants to connect to the Illinois Medicaid Fact Sheet on AAP.org, creating a more consistent URL is better for future-proofing. E.g.—

  • aap.org/en-us/Documents/federaladvocacy_medicaidfactsheet_illinois

Can easily be replaced year to year. Whereas

  • aap.org/en-us/Documents/federaladvocacy_medicaidfactsheet_2014_illinois

Will eventually result in a dead link. Although some content is archivable on AAP.org, filenames (for both PDfs and web pages) should be named such that being directed to them will result in the user getting the most recent and relevant information. Remember that everything on the site is crawled by search.

Note: Generally speaking, AAP.org does not make a very good archive for content. AAP.org is a publishing platform for the web and not necessarily an internal DMS (document management system), DAM (digital asset management system), or archiving entity. As a rule, PDFs should not be housed within the AAP.org CMS unless they are linked to directly within the external-facing content of AAP.org.

Optimizing in Adobe Acrobat

PDFs are often much larger than web pages. As such, they need to be "optimized" for the web to reduce the buren on our servers. The easiest method to optimize a PDF is with Adobe Acrobat.

  1. Open your PDF in Adobe Acrobat
  2. Select File > Save as Other... > Optimized PDF...
  3. Select optimize

If your PDF contains large, unoptimized images, you may want to adjust the image settings within the optimization pane before converting the file. Often the changes made by PDF optimization are barely visible to the user, whereas the reduction in file size is considerable. This is important when considering that PDFs are often larger than web pages.
