Versus DreamWeaver

Why is SharePoint "less flexible" than DreamWeaver?

SharePoint is less flexible than DreamWeaver, but that's because SharePoint and DreamWeaver are entirely different things.

SharePoint is a platform on which websites (intranets, extranet, and internet publishing sites [e.g., AAP.org, HealthyChildren.org]) can be built. DreamWeaver, however, is an IDE (integrated development network). It's a tool used to build websites and not necessarily manage them. Even an out-of-the-box installation of SharePoint comes preloaded with tools to managed tens of thousands (millions even) of documents. Further, it provides editorial with the ability to tag content within the CMS (content management system). As AAP.org continues to grow in scope, metadata becomes more and more important for all AAP.org properties. Metadata allows AAP.org Editorial to make considerable improvements to the site collection, including, but not limited to, the following:

  • Schedule content ahead of time according to an editorial calendar.
  • Promote results within search to make sure end-users are receiving the most appropriate content according to their search terms.
  • Generate content dynamically from within a page; e.g., rather than providing the end user with a list of links to PDFs, web parts within SharePoint can dynamically display relevant content according to the site's URL, the page location, etc and can display this same dynamic content across hundreds--or thousands--of landing pages and web pages.
  • Clean out old or redundant content. This is not a value judgement of content, since it is natural for content owners to forget what's been added to the CMS over time. SharePoint's CMS allows Editorial to scrub the top-level site collection of numerous documents so that end-users aren't inadvertently provided with inaccurate—or worse, outdated—information. It is important to note that even content that it no actively linked on public-facing pages is still crawled by search.

Layout and Design

So I don't have any flexibility when it comes to the presentation of my content on AAP.org?

Design elements within the AAP.org site collection are not within the editorial wheelhouse and were already decided during the original move to SharePoint in 2011. That said, you have options when it comes to the way your user will experience the content of your new site on AAP.org. Editorial is currently working with DoIT to add other site layouts and designs (e.g, IHCW, Bright Futures) to AAP.org's current collection.

Many of the design elements for the individual subsites on the ColdFusion platform were determined by individual content owners using a WYSIWYG (what you see is what you get) HTML editor within DreamWeaver. DreamWeaver is an excellent tool for designing small websites, but scalability is limited with ColdFusion.

Page layouts and designs in SharePoint are created via CSS (cascading style sheets), which live separate from the content itself. This helps to create uniformity across the many subsites housed within the AAP.org site collection.

Now that AAP.org is fully responsive, it is more important than ever to rely on web best practices w/r/t presentation and (aesthetic) style. WYSIWYG editors create inline styles, which only provide a short-term solution for minor style changes. However, inline styles take precedent over external stylesheets (i.e., they are closer to the bottom of the "cascade" in the cascading style sheet), so they cause considerable problems when large-scale redesigns need to be undertaken for enterprise-level web properties. An apropos example is the migration you're working on right now, which could have been done programmatically if the content had been created according to an open, unified standard.
